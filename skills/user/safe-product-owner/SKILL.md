## Primary Workflow

SAFe PM executes across eight operational phases.

### Phase 1 — AI Product Vision & Strategy
*Product vision, multi-PI roadmap, competitive intelligence, AI opportunity identification*

- Craft compelling AI product vision aligned with business strategy and market opportunity
- Build multi-PI roadmaps with clear AI capability milestones (model accuracy targets, latency thresholds, feature launches)
- Conduct AI product market analysis: LLM landscape, foundation model capabilities, competitive positioning
- Identify AI-driven innovation opportunities: automation potential, personalization ROI, prediction value
- Define responsible AI strategy: bias audit schedule, fairness metrics, explainability requirements

### Phase 2 — SAFe Backlog Management
*Program Backlog, WSJF prioritization, AI-specific Epic decomposition*

**WSJF for AI Products (Enhanced):**
```
WSJF = Cost_of_Delay / Job_Size
Cost_of_Delay = User_Business_Value + Time_Criticality + Risk_Reduction + AI_Complexity_Modifier
```

AI Complexity Modifier: +1 for novel ML approaches, +2 for foundation model fine-tuning, +3 for novel AI capability

**AI User Story Format:**
```
As a [user persona],
I want [AI capability],
So that [business outcome].

Acceptance Criteria:
- Model performance threshold: [metric ≥ threshold] (e.g., Precision ≥ 0.85)
- Latency requirement: [P95 response time < X ms]
- Fallback behavior: [graceful degradation when model confidence < threshold]
- Bias check: [protected attribute performance gap < Y%]
```

- Epic decomposition: Business Epic → Enabler Epic → Features → AI User Stories → Tasks
- Responsible AI checkpoint: ethics review embedded in Feature acceptance criteria

### Phase 3 — PI Planning for AI Products
*PI readiness, team capacity planning, AI-specific risk identification*

**Pre-PI Planning:**
- AI capability assessment: current model performance vs. PI target state
- Data readiness: training data availability, quality assessment, labeling capacity
- Infrastructure readiness: GPU/compute capacity, MLOps pipeline maturity
- Team capacity: AI engineer availability, data scientist allocation, compute budget

**During PI Planning:**
- Present AI product vision and business context (15 min max)
- Guide team breakouts: story planning with AI acceptance criteria
- Identify AI-specific risks: model performance below threshold, data drift, regulatory change
- Document AI enabler stories (infrastructure, experimentation, data pipeline)
- Validate PI Objectives: committed vs. uncommitted for AI features with high uncertainty

**PI Objectives for AI Products:**
- Committed: proven model capabilities with incremental improvements
- Uncommitted: experimental AI features with outcome uncertainty

### Phase 4 — Sprint Execution & Model Development
*Sprint planning, AI experiment management, model iteration*

- Sprint planning: balance feature development with model improvement stories
- AI experiment tracking: hypothesis → experiment design → training run → evaluation → decision
- Model iteration management: version control, experiment reproducibility, baseline comparison
- Data story management: data collection, labeling, validation, pipeline stories
- Definition of Done for AI: code merged, tests passed, model evaluated, bias check passed, documentation updated

### Phase 5 — AI Product Metrics & Analytics
*Model performance KPIs, business impact metrics, leading indicators*

**AI Product KPI Framework:**

| Layer | Metrics |
|-------|---------|
| Model Quality | Precision, Recall, F1, AUC-ROC, calibration, distribution shift |
| Product Performance | Feature adoption, task completion rate, user satisfaction (CSAT, NPS) |
| Business Impact | Revenue attributed, cost reduced, time saved, conversion lift |
| Responsible AI | Bias metrics by protected attribute, explainability coverage, refusal rate |
| MLOps Health | Data drift rate, model staleness, pipeline reliability, retraining frequency |

**A/B Testing for AI:**
- Bayesian A/B with proper stopping rules (no peeking problem)
- Multi-armed bandit for prompt variant optimization
- Holdout strategy for long-term model quality assessment

### Phase 6 — System Demo & Release Management
*AI feature demonstration, release planning, stakeholder communication*

- System Demo preparation: showcase AI capabilities with business outcomes, not technical specs
- Release readiness for AI: model performance validated, bias audit passed, rollback plan ready
- Feature flags for AI: gradual rollout with monitoring triggers for automatic rollback
- Stakeholder communication: translate model metrics to business language ("85% precision means 15% false alerts — we reduced alert fatigue by X%")

### Phase 7 — Responsible AI Governance
*Ethics checkpoints, fairness monitoring, compliance, explainability*

- Bias detection pipeline: evaluate model on protected attribute cohorts (gender, race, age, disability)
- Fairness metrics: demographic parity, equalized odds, individual fairness as appropriate
- Explainability framework: SHAP/LIME for feature importance, counterfactual explanations for regulated decisions
- Privacy-preserving ML: differential privacy, federated learning options for sensitive data
- AI incident response: detection, investigation, remediation, stakeholder communication protocol

### Phase 8 — Continuous Improvement & Model Evolution
*Retrospectives, model retraining triggers, product evolution*

- Sprint retrospectives for AI: what experiments failed/succeeded, process improvements
- Model retraining triggers: performance degradation threshold, data distribution shift, business rule changes
- Continuous learning: evaluate new foundation model releases for replacement/augmentation opportunities
- Customer feedback loop: systematic collection of AI output quality ratings for model improvement

## Constraints & Governance

### AI Product Ethics — Non-Negotiable

**Hard constraints:**
- No AI product ships without bias evaluation on protected attribute cohorts
- High-stakes AI decisions (employment, credit, healthcare) require human review capability — no fully autonomous AI for these domains
- Explainability required for regulated AI applications — black-box models not acceptable where explanations are legally required
- Model performance claims require validation data, not just training set metrics
- Responsible AI governance is not optional — it is embedded in Definition of Done for every AI feature
