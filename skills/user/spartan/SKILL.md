## Primary Workflow

SPARTAN executes the **7-Step SPARTAN Compression Methodology (SCM)** — identical structure for both input types; internal operations differ per path.

### Dual-Input Decision Matrix

| Signal | Route |
|--------|-------|
| Input contains `.md`, `.json`, `.docx` PDD structure | → PDD Compression Path |
| Input contains source files (`.ts`, `.py`, `.js`, `package.json`, etc.) | → Codebase Compression Path |
| Input is a GitHub URL | → Clone, detect → route to correct path |
| Input is both PDD and codebase | → Run PDD path first, cross-validate against codebase |
| User specifies override | → Follow user instruction |

### Step 1 — SCAN
Establish complete surface area of the input.

- **PDD:** Count prompts across all phases, map user-facing features, catalogue technology layers, record source metrics
- **Codebase:** Map all files/directories/entry points, identify exposed API endpoints, catalogue all dependencies, record LOC/module/service counts

**Output:** SPARTAN Input Manifest — complete source-state record.

### Step 2 — PROFILE (CLASS A / B / C Classification)

**Classification Test (applies to every prompt AND every code module):**
```
Q1: Does a user see, touch, or depend on this? → CLASS A (KEEP)
Q2: Does this serve system correctness (auth, data integrity, error handling)? → CLASS A
Q3: Can a platform-native tool replace this at MVP scale with no UX loss? → CLASS B (SYNTHESISE)
Q4: Does this serve scale (>500 MAU, >$10K MRR, enterprise compliance)? → CLASS C (DEFER)
```

| CLASS | Label | Action |
|-------|-------|--------|
| **A** | KEEP | Retained in MVP unchanged — the hard floor |
| **B** | SYNTHESISE | Replaced by platform-native equivalent — same result, simpler infrastructure |
| **C** | DEFER | Formally deferred with documented trigger — not deleted |

**Classification is irreversible within a single SPARTAN run.** Re-classification requires restarting from Step 2.

### Step 3 — ASSESS (VIBE DJ Integration)
Select the single VIBE app that will host all CLASS A features. VIBE DJ output is binding — SPARTAN cannot proceed to Step 4 without confirmed platform selection.

**Approved MVP VIBE App Candidates:**

| VIBE App | Best For | Monthly Cost | Deploy Time |
|----------|----------|--------------|-------------|
| Lovable | Full-stack web apps with Supabase backend | ~$50 | 3–5 days |
| Bolt.new | Rapid prototyping, UI-heavy apps | ~$30 | 1–3 days |
| v0.dev | Component libraries, design-first builds | ~$20 | 2–4 days |
| Replit | Backend-heavy apps, API services | ~$25 | 1–2 days |
| Cursor + Vercel | Full-stack with custom logic | ~$50 | 5–10 days |
| Firebase Studio | Mobile-first, Firebase-native apps | ~$40 | 3–7 days |

**Override rule:** If no single tool hosts all CLASS A features → GRO Containment escalation → recommend splitting into two SPARTAN runs.

### Step 4 — REDUCE
- **PDD:** Move CLASS C prompts to Upgrade Path Document; flag CLASS B for synthesis; lock CLASS A
- **Codebase:** Move CLASS C services to Deferred Services Registry; replace CLASS B with Stack Collapse equivalents; remove dead code entirely; log all removals in Dependency Collapse Map

### Step 5 — TRANSFORM
Rebuild input in MVP form using selected VIBE app's native patterns. Multiple CLASS B items serving the same function → collapse to ONE MVP prompt/module.

**Stack Collapse Examples:**

| Production | MVP Equivalent | Collapse |
|-----------|----------------|---------|
| Custom JWT auth (1,200 LOC) | Supabase Auth SDK (80 LOC) | 93% |
| 14 NestJS microservices (Docker/K8s) | 4 Supabase Edge Functions | 71% |
| PostgreSQL + PgBouncer + Redis | Supabase managed Postgres | CLASS B Synthesis |
| Multi-region CDN (CloudFront) | Vercel edge network | $500 → $20/mo |
| Monitoring stack (Datadog) | Supabase built-in logs | $300 → $0/mo |

### Step 6 — ZPOS+5 Application
Apply token/semantic optimization to all text content in the compressed output.

| Content Type | Method | Token Target | Semantic Floor |
|--------------|--------|-------------|----------------|
| Executive prompts (P0 phases) | PRISM | 35–40% | 97%+ |
| Technical implementation prompts | QUANTUM | 40–45% | 96%+ |
| Structured documentation prompts | SYNTHESIS | 38–42% | 96%+ |
| High-volume inline code comments | NEXUS | 45–50% | 95%+ |
| README and deployment guides | PRISM | 37–42% | 97%+ |

**Rule Z-04:** For codebase paths, ZPOS+5 applies ONLY to text content (comments, docstrings, README). Never modifies functional code logic.

### Step 7 — PACKAGE
Assemble, certify, and deliver all output artifacts.

**PDD Output Package:**
1. MVP ATLAS PDD (compressed, ZPOS+5-optimized, SPARTAN-certified)
2. Stack Collapse Map (production layers → MVP equivalents)
3. SPARTAN Compression Report (9-dimension CR metrics)
4. ZPOS+5 Optimisation Report (per-prompt token metrics)
5. Upgrade Path Document (CLASS C triggers + migration steps)
6. FORGE Certification Block

**Codebase Output Package:**
1. MVP Codebase (compressed, dependency-collapsed)
2. Dependency Collapse Map (production → MVP module equivalence)
3. SPARTAN Compression Report
4. ZPOS+5 Documentation Report (inline comment + README optimization)
5. Deployment Guide (single-developer step-by-step execution plan)
6. Upgrade Path Document (trigger-based production restoration plan)
