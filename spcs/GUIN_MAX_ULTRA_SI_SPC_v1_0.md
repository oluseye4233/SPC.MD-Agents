### 2.2 Executive Summary

GUIN MAX ULTRA SI is the world's first **codec-native universal matchmaking engine**. It solves a structural
problem that emerges the moment an ecosystem runs more than one platform: each platform develops its own
profiles, its own scoring, its own social graph — and the connections that could form *between* platforms never
form, because there is no shared language of compatibility.

The Junglenomics ecosystem already solved this language problem without naming it. Every deployable object the
ecosystem produces — a human's professional profile, a Super Prompt Card, a Molecular Agent, a CCGE scenario,
a marketplace listing, a job posting, a government RFP, a guild — is, or can be losslessly expressed as, a **card
encoded in the Junglenomics Primitive Card Codec (JPCC)**: the Atomic Prompt 7-tuple `(I, O, C, T, M, V, Δ)`
wrapped in the Context Craft 7-pillar specification, plus DISC profile, Camelot archetype, JST/O*NET vector, and
synergy/conflict descriptors. Because every entity carries the *same* codec, the compatibility between **any two
things the ecosystem can name** becomes a computable quantity — regardless of which platform each lives on.

GUIN MAX is the engine that reads that codec. It is the synthesis of two parent intelligences governed by a third:

- **GUIN+** contributes the **identity and social-graph DNA** — evidence-based profiles, the Neo4j connection
graph, evidence-gated endorsements, guilds, Knight Rank, and Pinecone-based mentor matching.
- **GAMEMXT** contributes the **matchmaking-matrix DNA** — compatibility scoring, multi-axis match filtering,
compatibility-radar visualization, match notifications, discovery feeds, and pairing-level anti-pattern detection.
- **SPHINX ULTRA SI** (Seat #0, Grand Architect) **governs the fusion** — resolving conflicts, enforcing the
JCSE quality floor, and binding the result into the 4J.BONSAI production floor.

The output is a single sovereign capability: **GUIN MAX takes any two JPCC-encoded cards and returns a match
verdict** — a Connection eXchange Score (CXS, 0–100), a synergy and conflict report, the recommended connection
type, and a platform-native action to enact it. It does this for humans seeking mentors, agents seeking synthesis
partners, talent seeking opportunity, tasks seeking the Molecular Agent that fits them, and players seeking
worthy opponents — all on the same engine, all in the same codec.

## ⬡ SECTION 4 — ARCHETYPE & THE PRIMITIVE CARD CODEC

### 4.1 Camelot Archetype — THE HERALD

GUIN MAX occupies the Herald archetype: the figure who travels between realms, speaks every court's language, and

### 4.2 The Junglenomics Primitive Card Codec (JPCC)

The codec is the foundational design building block on which every 4J.BONSAI platform is built. GUIN MAX reads
and writes it. A JPCC **Compatibility Card** normalizes any platform entity into a single comparable structure:

| Codec Field | Source Primitive | Role in Matching |
|-------------|------------------|------------------|
| **I — Input Contract** | Atomic Prompt tuple | What the entity needs / accepts (the "asks") |
| **O — Output Contract** | Atomic Prompt tuple | What the entity offers / produces (the "gives") |
| **C — Constraint Set** | Atomic Prompt tuple | Hard boundaries that gate any match (region, tier, ethics, availability) |
| **T — Test Signal** | Atomic Prompt tuple | Verified evidence (KCSE/JCSE history, test pass rate) — powers the evidence gate |
| **M — Metadata** | Atomic Prompt tuple | Owner, version, risk tier, JST/O*NET vector, JCSE score |
| **V — Validation** | Atomic Prompt tuple | Compatibility thresholds, semantic-preservation floor |
| **Δ — Change Log** | Atomic Prompt tuple | History; powers recency decay and trend signal |
| **7-Pillar Vector** | Context Craft | `[SYSTEM, ROLE, INSTRUCTION, EXAMPLE, CONSTRAINT, FORMAT, DATA]` embedded for vector affinity |
| **DISC** | Camelot Antology | Personality complementarity axis |
| **Archetype** | Camelot Antology | Role-fit and team-formation axis |
| **Synergy / Conflict descriptors** | JCSE Synergy Matrix | Named synergy multipliers and the 3 anti-pattern conflict classes |

**Encode / Decode.** The codec is bidirectional. The **encoder** projects a platform-native object (a GUIN+
profile, a SPHINX listing, a job posting, an MA) into a Compatibility Card. The **decoder** translates a match
verdict back into a platform-native action (a mentor-request, a synthesis-pair handoff, a job application, an MA
binding). This is what makes a *single* engine serve *every* platform.

### 4.3 The Gives/Asks Duality (O↔I Reciprocity)

The heart of matching is reciprocity: a strong match exists when one card's **Output (gives)** satisfies another
card's **Input (asks)**, *and* the reverse holds to a useful degree. Mentor↔mentee, SPC↔SPC synthesis, talent↔job,
and task↔MA are all special cases of O↔I reciprocity computed over the shared codec.
