### STANDARD DJG TEMPLATE — TIER A (Vercel + Railway + Supabase)

#### PHASE 0 — PRE-FLIGHT (30 minutes)
**Step 1: Requirements Audit**
- Confirm tech stack: frontend framework, backend runtime, database type
- Confirm compliance requirements (GDPR / HIPAA / None)
- Confirm target traffic volume and budget ceiling
- Confirm domain status (new domain / existing domain / subdomain)
- Output: **Hosting Requirements Profile (HRP)**

**Step 2: Account Setup**
- Create accounts: Vercel (vercel.com), Railway (railway.app), Supabase (supabase.com)
- Link all three to the same GitHub organization
- Enable two-factor authentication on all three platforms
- Save platform API keys to secure vault (1Password / Bitwarden)

**Step 3: Repository Preparation**
- Confirm monorepo vs. polyrepo structure
- Create `/apps/web` (frontend), `/apps/api` (backend) if monorepo
- Add `.env.example` file with all required environment variable keys (values excluded)
- Confirm all secrets are in `.gitignore` — NEVER commit `.env`

#### PHASE 1 — DATABASE FOUNDATION (45 minutes)
**Step 4: Supabase Project Initialization**
- Create new Supabase project in target region (closest to primary user base)
- Note: Project URL, Anon Key, Service Role Key (store securely)
- Enable Row Level Security (RLS) on all tables from day one

**Step 5: Database Schema Deployment**
- Run migration files via Supabase CLI: `supabase db push`
- Verify all tables created correctly in Supabase Studio
- Enable **Point-in-Time Recovery (PITR)** on Pro plan
- Configure **connection pooling** (PgBouncer) for backend connections

**Step 6: Authentication Configuration**
- Configure auth providers (Email/Password minimum; OAuth if required)
- Set redirect URLs to both local (`localhost:3000`) and production domains
- Configure email templates for verification and password reset

#### PHASE 2 — BACKEND DEPLOYMENT (60 minutes)
**Step 7: Railway Project Setup**
- Create new Railway project
- Connect to GitHub repository (backend folder or root)
- Configure **Root Directory** to `/apps/api` if monorepo

**Step 8: Environment Variables (Backend)**
- In Railway Variables panel, add all backend secrets:
  - `DATABASE_URL` (Supabase connection string — Transaction mode)
  - `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`
  - `CLAUDE_API_KEY` (if AI-powered)
  - All third-party API keys
- Set `NODE_ENV=production`

**Step 9: Backend Build & Deploy**
- Confirm `package.json` build script: `"build": "nest build"` (NestJS)
- Confirm `start:prod` script: `"start:prod": "node dist/main"`
- Railway auto-deploys on git push to `main`
- Verify deployment logs — look for `Application is running on port 3000`
- Test health endpoint: `https://[your-railway-url]/health` → `{ status: "ok" }`

#### PHASE 3 — FRONTEND DEPLOYMENT (45 minutes)
**Step 10: Vercel Project Setup**
- Import GitHub repository to Vercel
- Set **Framework Preset** to Next.js (or detected framework)
- Set **Root Directory** to `/apps/web` if monorepo
- Configure **Environment Variables** in Vercel project settings:
  - `NEXT_PUBLIC_API_URL` = Railway backend URL
  - `NEXT_PUBLIC_SUPABASE_URL` = Supabase project URL
  - `NEXT_PUBLIC_SUPABASE_ANON_KEY` = Supabase anon key

**Step 11: Domain Configuration**
- In Vercel: Add custom domain → enter your domain
- In DNS provider: Add CNAME record pointing to `cname.vercel-dns.com`
- Add A record for apex domain: `76.76.21.21`
- SSL/TLS: Auto-provisioned by Vercel via Let's Encrypt (wait 24–48 hours for DNS propagation)
- Add `www` redirect to apex domain or vice versa

#### PHASE 4 — CI/CD & MONITORING (30 minutes)
**Step 12: Pipeline Activation & Go-Live Verification**
- Confirm GitHub auto-deploy: push to `main` → Railway backend deploys → Vercel frontend deploys
- Create **staging branch** (`staging`/`develop`) → separate Railway staging environment
- Set up Vercel preview deployments for all PRs
- Install **Sentry** for error tracking (free tier sufficient at launch)
- Configure uptime monitoring: **Better Uptime** or **UptimeRobot** (free) — ping `/health` every 60 seconds
- Set up alert: email notification if uptime drops below 99%
- **LAUNCH CHECKLIST:**
  - [ ] Health endpoint returns 200
  - [ ] Auth flow works end-to-end
  - [ ] Database reads/writes confirmed
  - [ ] All environment variables loaded
  - [ ] Custom domain resolving with HTTPS
  - [ ] Error tracking active
  - [ ] Rollback procedure documented

### SDF ARTIFACT 2: GitHub Actions CI/CD Pipeline
```yaml
# HOST DJ — CI/CD Pipeline
# .github/workflows/deploy.yml

name: Deploy to Production

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    name: Test & Quality Gate
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run tests
        run: npm test
      
      - name: Run linting
        run: npm run lint
      
      - name: Build check
        run: npm run build
        env:
          # Dummy values for build-time env check
          NEXT_PUBLIC_API_URL: ${{ secrets.STAGING_API_URL }}
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.SUPABASE_ANON_KEY }}

  deploy-backend:
    name: Deploy Backend (Railway)
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      
      - name: Install Railway CLI
        run: npm install -g @railway/cli
      
      - name: Deploy to Railway
        run: railway up --service backend
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}

  verify-deployment:
    name: Verify Deployment Health
    needs: [deploy-backend]
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Health Check (Backend)
        run: |
          sleep 30
          response=$(curl -s -o /dev/null -w "%{http_code}" ${{ secrets.BACKEND_URL }}/health)
          if [ $response -ne 200 ]; then
            echo "Health check failed with status $response"
            exit 1
          fi
          echo "✅ Backend healthy"
      
      - name: Notify on success
        run: echo "🚀 Deployment successful — $(date)"
```
